TeeLab en crisis

La API sigue en pie, pero los usuarios notan lentitud extrema, operaciones que no terminan y bloqueos bajo carga simultánea. Tu misión: diagnosticar, medir y proponer soluciones justificadas.

Enunciado

Objetivo

Esta página es vuestro panel de diagnóstico del backend de TeeLab. Desde aquí podéis lanzar peticiones a los distintos endpoints, medir su comportamiento, comparar resultados y extraer conclusiones sobre los problemas de rendimiento.

La práctica tiene dos fases: primero exploráis sin mirar el código, observando síntomas. Después analizáis el código para entender las causas y proponer mejoras.

🔍 Fase 1 — Exploración ciega
  • Probad los endpoints desde esta interfaz o con Thunder Client.
  • Observad tiempos de respuesta, tamaño y errores.
  • Comparad entre endpoints: ¿cuál tarda más? ¿en qué condiciones?
  • Repetid las pruebas con varias peticiones en paralelo.
  • Anotad síntomas: lentitud, bloqueo, timeout, peso de respuesta.
🔬 Fase 2 — Análisis del código
  • Leed el código de cada servicio y proceso del backend.
  • Relacionad los síntomas observados con el código.
  • Identificad qué mecanismo usa cada endpoint: sync, worker, child, fork…
  • Proponed soluciones justificadas. No se trata de aplicar cluster a todo.
  • Explicad qué mejora tendría más impacto real en TeeLab.

2. Investigación

Análisis del código

Una vez tengáis datos, es el momento de abrir el código. El backend está en src/. Cada endpoint tiene su servicio, y cada servicio usa una estrategia diferente para gestionar la carga.

Problema A
/api/catalogo/catalogo-completo

El catálogo completo se genera mediante un proceso hijo con spawn. Observad cuánto tarda y si el resto de endpoints se ven afectados durante la generación.

Problema B
/api/pagos/finalizar

El endpoint de pagos puede operar en modo sync o worker. En modo sync, calcula una firma criptográfica que bloquea el event loop durante varios segundos. Comprobadlo.

Problema C
/api/tickets/:id/pdf

El ticket se genera vía spawn o exec según la variable de entorno PDF_MODE. La respuesta puede llegar a varios MB. ¿Influye el método en el comportamiento?

Problema D
/api/recomendaciones

Usa fork() con un timeout de 4 segundos. El cálculo es intensivo en CPU. ¿Qué pasa cuando llegan varias peticiones a la vez? ¿El fork protege el proceso principal?

Problema E
/health y /api/catalogo/camisetas

Estos endpoints deberían ser siempre rápidos. ¿Lo son cuando hay carga en los demás? Si se degradan, ¿a qué se debe? ¿Qué solución de arquitectura lo resolvería?

📋 Preguntas para el informe final
  1. ¿Qué endpoint os ha parecido más problemático y por qué?
  2. ¿Qué síntomas habéis observado: lentitud, bloqueo, timeout, gran peso de respuesta, errores?
  3. ¿Qué diferencia hay entre un problema de saturación global del event loop y uno localizado en un proceso hijo?
  4. ¿Cuándo usaríais worker_threads frente a child_process? ¿Y fork()? ¿Y cluster?
  5. ¿Qué solución aplicaríais a cada problema y por qué?
  6. ¿Qué mejora tendría más impacto real en el sistema de TeeLab con el mínimo cambio de código?

3. Implementación de Cluster

¿Cluster lo arregla todo?

Habéis diagnosticado los problemas, habéis propuesto soluciones. Ahora toca comprobar una de ellas en la práctica: aplicar cluster al servidor y volver a medir. La pregunta no es "cómo se implementa", sino "qué mejora y qué no mejora".

🔧 Lo que tenéis que hacer
  • Cread un fichero server.js en la raíz del proyecto (no toquéis app.js).
  • Usad el módulo cluster de Node.js para lanzar un worker por núcleo de CPU (os.availableParallelism() o os.cpus().length).
  • El proceso primario (primary) solo gestiona el ciclo de vida de los workers. Los workers importan y escuchan con app.
  • Arrancad con node server.js en lugar de node src/app.js. El endpoint /health ahora devolverá el pid del worker que responde — observadlo.
  • Repetid las mismas pruebas de la Fase 1: individual y bajo carga paralela.
💡 Pista de implementación
  • cluster.isPrimary distingue el proceso maestro de los workers.
  • El maestro hace cluster.fork() N veces y escucha 'exit' para relanzar workers caídos.
  • Los workers simplemente hacen app.listen(PORT) — Node reparte las conexiones entrantes automáticamente.
  • Aproximadamente 15–20 líneas de código. No es magia, es el patrón que visteis en los apuntes.
  • Añadid console.log con el PID en cada worker para ver el reparto en acción.
Observación 1

📋 Preguntas para el informe — Fase 3
  1. ¿Qué problemas de TeeLab resuelve cluster y cuáles no resuelve? Argumentad con los datos de vuestras mediciones.
  2. ¿Por qué el modo sync de pagos sigue siendo un problema incluso con cluster? ¿Qué número de peticiones simultáneas lo vuelve a saturar?
  3. ¿Qué ventaja tiene cluster sobre simplemente usar worker_threads para todo? ¿Son intercambiables o complementarios?
  4. ¿Qué pasaría si un worker se cae mientras procesa un pago? ¿Cómo lo gestionaría un cluster bien implementado?
